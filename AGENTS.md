# FretFlow AI Development Guide

## 最初に行うこと

1. `git status --short --branch` でブランチと未コミット変更を確認する。
2. ユーザーの変更を上書きしない。既存差分が対象ファイルと重なる場合は先に内容を確認する。
3. 毎セッションの開始時に `main` ブランチへ移動し、既存差分と衝突しないことを確認してから `git pull --ff-only origin main` を実行する。
4. `README.md`、このファイル、変更対象モジュール、対応する `test/*.test.mjs` を読む。
5. 実装前に `npm test` を実行してベースラインを確認する。

## アーキテクチャ

- `index.html`: 静的な画面構造。インラインJavaScriptを置かない。
- `app.js`: DOM操作、イベント、機能モジュールの統合。音楽理論やファイル形式のロジックを追加しない。
- `js/music-data.js`: 音名、対応コード種別、チューニング、既知のフォーム。対応コード追加時の単一情報源。
- `js/chords.js`: 入力の厳密な字句解析、正規化、構成音、カポ移調。入力全体を必ず消費し、推測補正しない。
- `js/voicings.js`: 運指候補、妥当性、コスト、動的計画法による最適化。UIに依存させない。
- `js/exporters.js`: MIDIとMusicXMLの純粋な生成処理。DOMやBlob URL管理を入れない。
- `test/`: ソースモジュールと対応する名前でNode標準テストを置く。
- CSSは共通デザインを`styles.css`、譜面を`score.css`、追加UIを`ui.css`に置く。

## 実装規則

- ビルド工程と外部依存を増やさず、ブラウザ標準ES ModulesとNode標準APIを優先する。
- ビジネスロジックは引数を受けて値を返す純粋関数にし、`app.js`からexport可能な機能モジュールへ移す。
- 対応外入力を近いコードへ変換したり、認識可能な部分だけ採用したりしない。`{names:[], error:string}`で明示的に失敗させる。
- コード品質を追加するときは`music-data.js`の`CHORD_TYPES`、`chordInfo`、運指生成、コード一覧、テストを一緒に更新する。
- 表示するユーザー入力はHTMLエスケープする。XML出力も専用エスケープを通す。
- MIDIはバイト列、MusicXMLは文字列を返し、ダウンロードURLの生成・破棄はUI層で行う。
- 既存公開関数の返り値形状を変える場合は、呼び出し元とテストを同じ変更で更新する。

## テスト規則

- 変更前後に`npm test`、コミット前に`node --check app.js`と`git diff --check`を実行する。
- 各公開関数に正常系、異常系、下限・上限・切替点の境界値を用意する。
- 不具合修正は、修正前に失敗する最小の回帰テストを追加してから実装する。
- 文字解析では空入力、連結入力、全区切り、異名記号、全対応品質、未対応品質、末尾ゴミを検証する。
- 運指では0件、1件、複数件、候補なし、フォーム・難易度・ポジション設定を検証する。
- MIDIでは127/128、16383/16384など可変長境界とチャンク長を検証する。
- MusicXMLでは小節数、音高、TAB弦/フレット、カポ、特殊文字エスケープを検証する。
- 時刻、乱数、DOM、ネットワークに依存するテストを避け、再現可能にする。

## 完了条件

- 要求した動作と回帰テストが実装されている。
- `npm test`、構文検査、`git diff --check`が成功する。
- READMEやこのガイドと実装に矛盾がない。
- `git status`に意図しないファイルがない。
- 作業完了後は変更をコミットし、`main` ブランチへpushしてリモートSHAを確認する。
